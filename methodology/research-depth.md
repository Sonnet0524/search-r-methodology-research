# 研究深度定义

> Research Agent应该追求的研究深度标准

**版本**: v1.0 | **更新**: 2026-03-07

---

## 📊 研究深度层次

### Level 0: 第一性原理

**问题**: XXX的本质是什么？为什么要XXX？

**方法**: 追问"为什么"直到无法再追问

**判断标准**:
- 是否理解根本原因？
- 能否回答"为什么要这样设计"？
- 是否触及问题的本质？

**示例**:
```
问题：为什么要使用文档化交互？

Level 0回答：
- 因为Agent需要异步协作
- 因为需要可追溯性
- 因为需要可自动化
→ 本质：解决Agent协作的同步阻塞问题
```

---

### Level 1: 理论框架

**问题**: XXX如何建模？决策模型是什么？

**方法**: 构建模型、形式化表达、逻辑推演

**判断标准**:
- 是否建立了理论模型？
- 能否指导具体设计？
- 逻辑是否自洽？

**示例**:
```
问题：文档化交互如何建模？

Level 1回答：
- Agent作为生产者/消费者
- 文档作为消息队列
- 文档状态作为交互协议
→ 模型：生产者-文档-消费者模型
```

---

### Level 2: 设计原则

**问题**: 应该遵循什么原则？原则如何权衡？

**方法**: 从理论推导原则

**判断标准**:
- 是否明确了设计原则？
- 能否指导具体方案选择？
- 原则之间是否有优先级？

**示例**:
```
问题：文档化交互应该遵循什么原则？

Level 2回答：
- 分层原则：Level 0/1/2按需加载
- 单向依赖原则：避免循环依赖
- 明确归属原则：每个文件有明确责任人
→ 原则：Context最小化原则
```

---

### Level 3: 实现思路

**问题**: 如何设计？有哪些方案？

**方法**: 从原则推导思路

**判断标准**:
- 是否提供了清晰的实现路径？
- 是否考虑了多种方案对比？
- 是否明确了权衡点？

**示例**:
```
问题：如何实现文档化交互？

Level 3回答：
- 方案1：手动更新文档
- 方案2：半自动化脚本
- 方案3：完全自动化（Message Queue）
→ 思路：先手动，再半自动，最后全自动
```

---

### Level 4: 具体实现 ❌

**问题**: 代码怎么写？配置放哪里？

**方法**: **这不是Research Agent的职责范围**

**判断标准**:
- Research Agent不应该停留在Level 4
- 应该由执行层Agent（如Data Agent、Template Agent）完成
- Research Agent只提供Level 0-3的深度

---

## 🎯 Research Agent的目标深度

### 标准要求

```
最小深度：Level 1（理论框架）
标准深度：Level 2（设计原则）
优秀深度：Level 0-2全覆盖
```

### 深度检查

**每次研究后自问**：
- [ ] 我是否理解了"为什么"？（Level 0）
- [ ] 我是否建立了理论模型？（Level 1）
- [ ] 我是否明确了设计原则？（Level 2）
- [ ] 我是否提供了实现思路？（Level 3）
- [ ] 我是否陷入了实施细节？（Level 4 ❌）

## 📈 深度递进示例

### 案例：质量门控研究

#### Level 0: 第一性原理

```
问题：为什么需要质量门控？

回答：
- 因为Agent需要知道何时该呼叫Human
- 因为Human参与需要明确的触发条件
- 因为需要平衡Agent自主性和Human控制

本质：质量门控是Human介入的触发器
```

#### Level 1: 理论框架

```
问题：质量门控如何建模？

回答：
- 三维评估模型：
  - 确定性维度（HIGH/MEDIUM/LOW）
  - 可接受性维度（HIGH/MEDIUM/LOW）
  - 混淆维度（有/无）
  
- 决策模型：
  IF 确定性=HIGH AND 可接受性=HIGH AND 无混淆
  THEN Agent自主决策
  ELSE 呼叫Human
```

#### Level 2: 设计原则

```
问题：质量门控应该遵循什么原则？

回答：
- PM自主性优先原则
- Human参与最小化原则
- 明确触发条件原则
- Token效率原则
```

#### Level 3: 实现思路

```
问题：如何实现质量门控？

回答：
- 方案1：在AGENTS.md中定义（当前）
- 方案2：提取为Skills（未来）
- 方案3：动态配置（长期）

权衡：Token占用 vs 灵活性
```

## 🔍 深度评估

### 自我评估清单

**Level 0评估**：
- [ ] 能回答"为什么"
- [ ] 理解根本原因
- [ ] 触及问题本质

**Level 1评估**：
- [ ] 建立了理论模型
- [ ] 模型逻辑自洽
- [ ] 能指导设计

**Level 2评估**：
- [ ] 明确了设计原则
- [ ] 原则可操作
- [ ] 原则有优先级

**Level 3评估**：
- [ ] 提供了实现思路
- [ ] 考虑了多种方案
- [ ] 明确了权衡点

**Level 4警示**：
- [ ] ❌ 我是否在写具体代码？
- [ ] ❌ 我是否在讨论配置细节？
- [ ] ❌ 我是否应该让执行层Agent来处理？

## 💡 深度提升技巧

### 技巧1: 持续追问

```
陈述：需要质量门控
追问：为什么需要？→ 因为需要判断何时呼叫Human
追问：为什么需要判断？→ 因为要平衡自主性和控制
追问：为什么需要平衡？→ 这是Agent系统的本质需求
→ Level 0达成
```

### 技巧2: 建模思维

```
观察：质量门控包含确定性、可接受性、混淆判断
建模：三维评估模型
形式化：IF ... THEN ... ELSE ...
→ Level 1达成
```

### 技巧3: 原则提取

```
观察：质量门控在不同场景下表现不同
提取：PM自主性优先原则
应用：在确定性HIGH时PM自主决策
→ Level 2达成
```

### 技巧4: 方案对比

```
问题：如何实现质量门控？
对比：
- 方案A：硬编码在AGENTS.md
- 方案B：提取为Skills
- 方案C：动态配置
权衡：Token效率 vs 灵活性
→ Level 3达成
```

## 🎯 实践指南

### 研究开始时

1. **明确研究问题**
2. **确定目标深度**（通常是Level 0-2）
3. **规划研究路径**

### 研究过程中

1. **持续评估当前深度**
2. **使用追问技巧提升深度**
3. **避免陷入Level 4细节**

### 研究结束时

1. **检查是否达到目标深度**
2. **补充缺失的深度层次**
3. **产出标准化的研究成果**
